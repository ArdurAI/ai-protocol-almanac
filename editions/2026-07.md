## MCP Security — The Slowest Moving Crisis in AI Infrastructure

### The state of the category

The MCP security crisis identified in the founding edition has not been resolved. If anything, the June 2026 data shows the problem is deeper than initially understood. The November 2025 spec mandates OAuth 2.1 with PKCE for remote MCP servers, but the client ecosystem is not implementing it.

### MCP OAuth refresh-token support matrix (June 2026)

| Support Level | Count | Change from April | Notable Clients |
|---------------|-------|-------------------|-----------------|
| Full support | 1 | +1 | Gemini CLI (PR #26312) |
| Partial / broken | 10 | +3 | Claude Desktop, Cursor, LibreChat, Zed, Goose, Cline, Continue.dev, Open WebUI, Copilot CLI, Windsurf |
| Not implemented | 2 | -4 | Claude.ai (Web), Amazon Q CLI |
| Unknown | 1 | — | — |

**Key finding**: Only **1 out of 14 major MCP clients** has fully working OAuth refresh-token support. Ten clients have partial implementations with known bugs (race conditions, token store sync failures, metadata discovery path errors). Two clients have not implemented refresh tokens at all.

### SEP-2207: The specification exists but the ecosystem isn't consuming it

- **SEP-2207** (OIDC-Flavored Refresh Token Guidance) was accepted as a Standards Track proposal on **February 4, 2026**
- **TypeScript SDK** merged SEP-2207 on **March 30, 2026**
- **Rust SDK** merged on **March 24, 2026**
- **Python SDK** PR #2039 remains open as of June 2026 with no activity since March 31
- **Most clients have not consumed the SDK updates** — the client applications (Claude Desktop, Cursor, VS Code) are not upgrading to the SDK versions that support refresh tokens

### CVE-2026-8719: The MCP OAuth privilege escalation vulnerability

- **CVSS 8.8** (High severity)
- **WordPress AI Engine plugin** (version 3.4.9) — authenticated Subscriber could escalate to MCP administrator via OAuth token manipulation
- **Patched** in AI Engine 3.5.0 (May 16, 2026)
- **Root cause**: Missing capability checks at token issuance and token validation time
- **Significance**: This is the first CVE specifically targeting MCP's OAuth implementation, and it surfaced in a widely-deployed plugin ecosystem (WordPress)

### The fix is three-part, and only one part is working

| Part | Status | What it needs |
|------|--------|---------------|
| **Spec** | ✅ Done | OAuth 2.1 + PKCE mandated in Nov 2025 spec; SEP-2207 accepted Feb 2026 |
| **SDKs** | ⚠️ Partial | TypeScript and Rust SDKs merged; Python SDK PR stale; most clients not upgrading |
| **Clients** | ❌ Broken | 1/14 fully working; 10/14 partial; 2/14 not implemented |

**Verdict**: The specification is sound. The SDK implementations are mostly there. The client applications are not adopting the SDK updates. The MCP security crisis is a **client adoption problem**, not a specification problem.

### Recommendations for the ecosystem

1. **MCP server implementors**: Accept longer access token TTLs (1-8 hours) as a temporary concession until client support improves. Document `mcp-remote` as the recommended workaround for broken clients.
2. **MCP client vendors**: Upgrade to the latest SDK versions (TypeScript ≥ post-March-2026, Rust ≥ post-March-2026). The Python SDK PR #2039 needs community attention.
3. **Registry operators**: Smithery, Glama, mcp.so, PulseMCP — add auth verification badges. Flag unauthenticated servers. Don't just list; validate.
4. **Gateway vendors**: Portkey, Cloudflare, Gravitee — continue adding auth layers, but the ultimate fix is at the server level, not the proxy level.

## A2A v1.0 — The Agent-to-Agent Protocol Matures

### Key developments since June

- **A2A v1.0 shipped at Cloud Next '26** (April 2026) with first-class Vertex AI support
- **Added features**: gRPC transport, signed Agent Cards (JWS), multi-tenancy
- **60+ organizations** in the AP2 payments extension
- **One-year milestone** celebrated April 9, 2026 — signals maturity inflection point
- **IBM Think 2026** confirmed convergence work on a unified "agent card" format across MCP and A2A

### The MCP + A2A convergence signal

IBM confirmed at Think 2026 that A2A and MCP are being unified around a single **"agent card" format** — one standard way to describe an entity whether it's a tool (MCP) or an agent (A2A). The Linux Foundation's Agentic AI Foundation is coordinating this convergence. By end of 2026, the tooling distinction may blur, but the architectural principles will remain: MCP for vertical tool access, A2A for horizontal agent coordination.

**Roster impact**: A2A remains Tier A. The v1.0 release solidifies its position. The convergence work is a positive signal for interoperability but does not change the current evaluation.

- **AutoGen + Semantic Kernel** merged into one coherent SDK

- **.NET and Python** — feature parity across both languages
- **Azure-native** — hooks into Azure OpenAI, Azure AI Search, Azure Functions

**Roster impact**: **New entrant, Tier B**. Strong enterprise positioning but Azure-centric. Added to roster as `microsoft-agent-framework.md`.

### Claude Managed Agents — Anthropic's managed runtime (Beta, April 2026)

Anthropic's fully managed agent runtime:

## IETF Standards — Draft Density Accelerates

Four new IETF drafts published in Q2 2026, bringing the total active agent-identity-relevant drafts to **8**:

| Draft | Title | Date | Status |
|-------|-------|------|--------|
| draft-klrc-aiagent-auth-01 | AI Agent Authentication and Authorization | Q1 2026 | Active; `agent_assertion` grant type; updated to -01 |
| draft-ni-wimse-ai-agent-identity-02 | WIMSE Applicability for AI Agents | March 2026 | Active; extends WIMSE to AI agents specifically |
| draft-yl-agent-id-requirements-00 | Digital Identity Management for AI Agent Communication Protocols | Q2 2026 | New; proposes globally unique agent identifier format |
| draft-aip-agent-identity-protocol-00 | Agent Identity Protocol (AIP) | Q2 2026 | New; verifiable identity and policy enforcement for AI agents |
| draft-nemethi-aid-agent-identity-discovery-00 | Agent Identity and Discovery (AID) | Q2 2026 | New; DNS-first agent discovery approach |
| draft-ietf-oauth-transaction-tokens-for-agents-04 | Transaction Tokens for Agents | Q2 2026 | Active; OAuth transaction tokens for agent workloads |
| draft-ietf-wimse-arch-07 | WIMSE Architecture | March 2026 | Active; includes AI agents as explicit use case |
| draft-abbey-scim-agent-extension-00 | SCIM for Agents | Q1 2026 | Active; `Agents` and `AgenticApplications` resource types |

**Critical observation**: No single draft spans the full agent identity lifecycle. The gaps cluster at authorization, delegation, and cross-domain trust — the hardest parts of the problem. The production ecosystem is still building on OAuth 2.1 + PKCE + RFC 8693 because those are the most mature standards available, regardless of whether they are the theoretically optimal primitives for agent identity.

**Roster impact**: WIMSE, AIMS, and SCIM for Agents remain Tier A. No changes to tier assignments. The draft density is positive but does not change the maturity assessment.

## Cross-standard findings

### The MCP security crisis is still everyone's problem

The lack of authentication in MCP servers continues to affect the entire ecosystem:
- **Registries** host unauthenticated servers with no verification badges
- **Gateways** add auth layers but cannot fix the underlying server problem
- **SDKs** integrate MCP but do not enforce auth
- **Security tools** scan but do not fix the root cause

The fix requires: spec enforcement + SDK adoption + client upgrades + community pressure. One month later, only the spec is fully ready.

### The MCP + A2A convergence is real but not yet actionable

IBM confirmed convergence work on a unified "agent card" format. The Linux Foundation's Agentic AI Foundation is coordinating. This is a positive signal for 2027-2028 but does not change the current architecture decision: MCP for tools, A2A for agents.

### The SDK landscape is consolidating around primitives

Four major SDK releases in Q2 2026 show convergence on the same primitives:
- **Sandboxed execution** (OpenAI, Microsoft, Claude Managed)
- **MCP native support** (Vercel, OpenAI, Microsoft, CrewAI)
- **A2A task delegation** (Microsoft, CrewAI)
- **Multi-model support** (Vercel 25+, OpenAI 100+, Microsoft Azure OpenAI)

The question is no longer "which primitives do I need?" but "which SDK bundles the primitives I need with the right vendor alignment?"

### The auth stack is getting more drafts but no more production-ready

Eight IETF drafts are active, but the production stack remains: OAuth 2.1 + PKCE for authorization, WIMSE for workload identity, and AIMS for agent identity. The drafts are converging on concepts but not on implementations.
