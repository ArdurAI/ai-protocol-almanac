## Context & Protocols — MCP Dominates, Security Lags

### The state of the category

The protocol landscape is converging around **MCP** as the de facto standard for agent-to-tool communication, with **A2A** emerging as the leading agent-to-agent coordination protocol. The **UCP** (Google + Shopify) and **x402** (Coinbase) are building the commerce and payment layer for autonomous agents. Meanwhile, **IETF standards** (WIMSE, AIMS, SCIM for Agents) are drafting the infrastructure for agent identity, auth, and lifecycle management—but are 2-4 years from RFC status.

The most critical finding: **MCP security is immature**. Despite the Nov 2025 spec mandating OAuth 2.1 for remote servers, Knostic's scan of ~2,000 MCP servers found **100% lacked authentication**. This is a systemic security failure in the most widely adopted protocol.

### Findings

- **MCP security is critically immature**: 100% of ~2,000 scanned MCP servers lacked authentication; 30+ CVEs Jan-Feb 2026. The spec mandates OAuth 2.1 but the ecosystem is not implementing it. This is the single biggest risk in the protocol landscape.

- **A2A adoption is accelerating**: 150+ organizations in production, with the ACP merger (August 2025) consolidating the REST-native agent communication space into A2A. SSE streaming is the most powerful feature but also the most fragile.

- **Commerce layer emerging**: UCP + x402 + AP2 + MPP are building a four-layer commerce stack (product discovery, cart, checkout, payment). UCP handles the commerce layer; x402/AP2/MPP handle the payment layer. The separation is by design but adds complexity.

- **IETF standards urgency**: AIMS, WIMSE, and SCIM for Agents were all drafted in Q1 2026 but are 2-4 years from RFC status. Meanwhile, the ecosystem needs these standards now. The gap between draft and production is being filled by vendor-specific solutions (Portkey, Stytch, Scalekit, Aembit).

- **AI gateways becoming standard infrastructure**: 70% projected adoption by 2028. Portkey leads with MCP Gateway GA (Jan 2026) and open-source core (March 2026). Google Apigee, Gravitee, and Cloudflare are targeting enterprise.

- **Registry explosion**: 5 major MCP registries with 50,000+ servers combined. Smithery (3,305+), Glama (22,000+), mcp.so (17,186+), PulseMCP (6,970+), MCPize (500+). The registry space is consolidating around discovery, hosting, and monetization.

- **Secrets management duality**: HashiCorp Vault (open source + enterprise, self-hosted) vs. AWS Secrets Manager (managed, AWS-only). For multi-cloud AI strategies, Vault is the only viable option. For AWS-native, Secrets Manager is simpler but more expensive at scale.

- **Security tools are reactive, not preventive**: Knostic (scanning), DataDome (protection), and Invariant Labs (research) are all addressing the MCP security crisis. But the root cause is the spec/auth gap, not a lack of security tools.

- **Trust and verification nascent**: Trulioo KYA (Digital Agent Passports) and Visa TAP (Trusted Agent Protocol) are early entrants in the "Know Your Agent" space. This is the regulatory frontier—expect EU AI Act and other regulations to drive adoption.

- **On-chain identity experimental**: ERC-8004 (mainnet Jan 2026) is the first on-chain agent identity and reputation standard. It is niche but significant for decentralized agent ecosystems.

- **SDK landscape**: Three dominant SDKs (Vercel AI SDK, OpenAI Agents SDK, Anthropic SDK) serve different use cases: multi-provider (Vercel), OpenAI-native (OpenAI), Claude-native (Anthropic). There is no clear winner—each has distinct strengths and vendor lock-in tradeoffs.

## Cross-standard findings

### The MCP security crisis is everyone's problem

The lack of authentication in MCP servers affects not just MCP but the entire ecosystem:
- **Registries** (Smithery, Glama, mcp.so) host unauthenticated servers
- **Gateways** (Portkey, Cloudflare) are adding auth layers but can't fix the underlying servers
- **SDKs** (Vercel, OpenAI, Anthropic) integrate MCP but don't enforce auth
- **Security tools** (Knostic, DataDome) are scanning but not fixing

The fix requires: spec enforcement + tooling + community pressure. The spec is there; the implementation is not.

### The four-layer commerce stack

Agent-driven commerce is emerging as a four-layer architecture:

| Layer | Protocol | Role | Status |
|-------|----------|------|--------|
| **Product Discovery** | UCP | Find products, compare prices, read reviews | Production (20+ retailers) |
| **Cart & Checkout** | UCP | Manage cart, obtain consent, finalize order | Production |
| **Payment Authorization** | AP2 | Cryptographically signed payment mandates | Early (Google ecosystem) |
| **Payment Execution** | x402 / MPP | Execute micropayments (crypto or fiat) | Production (x402: 150M+ txns) |

The layers are interoperable but not fully integrated. UCP checkout can trigger x402 payments, but the integration is not standardized. MPP (Stripe) is the fiat bridge.

### The auth stack: WIMSE + AIMS + OAuth 2.1

The emerging authentication stack for AI agents is a three-layer cake:

| Layer | Standard | Role | Maturity |
|-------|----------|------|----------|
| **Workload Identity** | WIMSE (IETF) | Prove the agent is a real workload on a real platform | Draft (RFC 2027-2028) |
| **Agent Authentication** | AIMS (IETF) | Prove the agent's identity and capabilities | Draft (RFC 2028+) |
| **Application Authorization** | OAuth 2.1 + PKCE | Authorize specific actions with user consent | Mature (IETF standard) |

This stack is complementary: WIMSE attests the platform → AIMS asserts the agent's identity → OAuth 2.1 authorizes the action. No single standard does all three.

### The observability backbone: OpenTelemetry GenAI

OpenTelemetry GenAI is becoming the universal observability layer:
- **SDKs** (Vercel, OpenAI, Anthropic) export `gen_ai.*` traces
- **Gateways** (Portkey) aggregate traces across providers
- **Platforms** (Langfuse, Langsmith, Datadog, New Relic) consume and analyze traces
- **Protocols** (MCP, A2A) can be instrumented with GenAI conventions

This is the most mature cross-standard integration in the ecosystem.

## Quest diary — June 2026

The founding edition of the Context Protocols, Authentication & Integration Almanac was built using a **deep research swarm** of 8 parallel agents conducting 112+ independent searches across the protocol landscape. The research process included:

1. **Discovery**: Wide search for protocols, standards, SDKs, registries, gateways, security tools, and auth platforms in the AI agent ecosystem.

2. **Triage**: 46 tools met the criteria (seriousness, activity, documentation, accessibility, scope). 30+ tools were rejected (toy specs, no implementations, abandoned projects, re-brandings with no divergence).

3. **Roster construction**: Machine-readable catalog in `data/roster.json` with 46 tools across 3 tiers.

4. **Deep reference writing**: 15 Tier A standards received deep-reference pages covering spec summary, version history, transport/schema details, known implementations, interoperability notes, compliance checklist, known issues, and benchmark results.

5. **Tool page generation**: 46 per-tool pages generated in `tools/` with overview, description, benchmark status, interoperability notes, and links to deep references.

6. **Edition writing**: This edition—landscape at a glance, per-category findings, cross-standard analysis, and Quest diary.

- **Standard deep references**: 15 Tier A standards were researched and documented with spec details, version history, interoperability matrices, and compliance checklists.

- **Cross-standard analysis**: Four major cross-standard patterns were identified: MCP security crisis, four-layer commerce stack, three-layer auth stack, and OpenTelemetry GenAI observability backbone.
